# Eidos Next.js 架构原则

本文档约束 `C:\Users\zhouh\Desktop\Eidos` 后续的 Next.js 改造方式。目标不是追求目录好看，而是让页面、服务端逻辑、共享规则和本地持久化边界清楚，降低继续开发时的耦合和回归风险。

## 1. 总原则

1. **Server First**
   - 能在服务端做的事情，优先在服务端完成。
   - 首屏取数、聚合、权限判断、配置解析、文件/数据库访问，不要默认放到客户端。

2. **BFF 架构**
   - Next.js 继续承担 Web UI + API/BFF。
   - `app/api/**/route.ts` 只做 HTTP 入站适配，不承载核心业务。
   - 服务端内部不要为了复用逻辑去 `fetch` 自己的 `/api/*`。

3. **Presentation / Feature / Server 分层**
   - `app/**/page.tsx`：页面壳、布局、组件拼装。
   - `features/**`：页面级状态编排、交互流程、派生逻辑、feature hook。
   - `server/**`：服务端用例、持久化、上游适配、本地文件与数据库访问。

4. **单一事实源**
   - 枚举、默认值、错误语义、时间/额度格式化、参数映射只能有一份事实源。
   - 禁止同一规则在页面、route、server、desktop 中分别重写。

5. **渐进式重构**
   - 先把边界立起来，再决定是否继续细拆。
   - 默认优先抽出一层 `view-model` 或 `hook`，不要一开始就拆成很多小组件或很多 util。

## 2. 目录职责

### `src/app`

- 只放路由、页面、layout、loading、error 和页面内少量展示组件。
- 页面文件不直接承载复杂请求编排、数据清洗、状态机。
- 真实业务页面默认采用 Server Component 外壳：
  - `page.tsx` 负责服务端取数、路由级动态策略和 client 组件拼装；
  - 交互 UI 放同目录 `*-client.tsx`；
  - 不在 `page.tsx` 顶部写 `"use client"`，除非页面确实没有任何服务端收益。
- 需要共享导航、启动刷新、应用壳的页面放到 `src/app/(app)` route group；根 `layout.tsx` 只保留全局 providers / toaster / html body 级能力。

### `src/features`

- 放每个功能自己的编排层。
- 典型内容：
  - `use-xxx-page.ts`
  - `xxx-view-model.ts`
  - 少量功能内共享 helper
- 这里可以依赖 `lib/api`、`store/*`、`shared/*`，但不直接读写 SQLite 或本地文件。

### `src/server`

- 放真正的服务端能力。
- 包括：
  - 应用用例编排
  - SQLite / 文件存储
  - 上游 ChatGPT / OpenAI 适配
  - CPA 同步
  - recover / image task / account pool 之类的后台逻辑

### `src/shared`

- 放纯规则、纯映射、纯 schema、纯 contracts。
- 要求可脱离 React、Next、Node 运行。

### `src/components`

- 放跨页面复用 UI，或体量较大的独立展示组件。
- 如果一个组件内部交互复杂，允许配套 `features/**/use-xxx.ts`，组件本身只保留展示和事件绑定。

### `src/lib/api`

- 按领域拆分前端 API client，而不是把 accounts、config、images、sync、version、requests 全堆在一个文件。
- `src/lib/api.ts` 只作为兼容聚合 facade。
- 共享 contract 类型优先集中到 `src/lib/api/types/index.ts`，避免 feature 和 server 侧从聚合入口顺手依赖不相关请求实现。

## 3. 页面与组件的写法

### 页面

页面优先采用下面的结构：

1. `page.tsx` 只负责：
   - 在服务端直接调用 `server/**` service / repository 读取首屏数据；
   - 设置必要的 `dynamic = "force-dynamic"`，避免本地 SQLite / 文件 / 运行态数据被 build-time 静态化；
   - 把初始数据传给同目录 `*-client.tsx`。

2. `*-client.tsx` 负责：
   - 调用 feature hook；
   - 渲染 JSX；
   - 绑定事件和局部展示状态。

3. 如果页面超过合理体量：
   - 第一刀先抽 `*-view-model.ts`
   - 第二刀再抽 `use-*-page.ts`
   - 第三刀才考虑拆表格区、侧边栏、对话框区等子组件

当前标准落地形态：

```text
src/app/(app)/accounts/page.tsx          # Server Component
src/app/(app)/accounts/accounts-client.tsx
src/app/(app)/image/page.tsx             # Server Component
src/app/(app)/image/image-client.tsx
src/app/(app)/requests/page.tsx          # Server Component
src/app/(app)/requests/requests-client.tsx
src/app/(app)/settings/page.tsx          # Server Component
src/app/(app)/settings/settings-client.tsx
```

### 大型交互组件

像图片编辑、工作台、复杂弹窗这类组件，默认采用：

- 组件文件：负责 JSX、DOM 结构、视觉和基础事件绑定
- `features/**/use-xxx.ts`：负责
  - 本地状态
  - effect
  - 请求编排
  - 提交前数据构造
  - 复杂交互逻辑

不要把缩放、拖拽、撤销重做、遮罩生成、提交流程全部塞在一个 `.tsx` 里长期增长。

## 4. API 与服务端规则

### Route Handlers

- 只做：
  - 参数读取
  - 基本校验
  - 调用 server service
  - HTTP 响应转换

- 不做：
  - 复杂业务规则
  - 多层重试策略
  - 大段持久化细节
  - 共享规则再实现一遍

- JSON body 必须经过运行时校验：
  - 使用 `src/server/request-validation.ts` 中的 `parseJsonBody(...)`；
  - 不允许在 route handler 里继续写 `await request.json() as SomeType`。

- Route Handler 的定位：
  - 公开 HTTP 端点；
  - OpenAI 兼容 `/v1/**` 入口；
  - multipart/file upload；
  - 长任务 / 外部系统调用；
  - 客户端显式刷新 fallback。

### Server Actions

- 页面内简单 mutation 优先考虑 Server Action。
- 典型适用场景：表单保存、单一配置写入、无需 multipart / stream / 外部兼容协议的操作。
- 已落地标准样例：`src/app/(app)/settings/actions.ts`。
- 不强行把所有写操作迁成 Server Action：
  - 图片生成 / 编辑 / 放大继续走 Route Handler，因为它们涉及 OpenAI 兼容协议、文件上传、长任务和外部 API 语义；
  - 账号导入继续走 Route Handler，因为它是 multipart 文件上传入口；
  - 对外暴露的 `/api/**` 和 `/v1/**` 不因页面内 Server Action 而删除。

### Server Services

- 继续按职责收口：
  - `account-*`
  - `image-*`
  - `cpa-sync-*`
  - `providers/*`
  - `*-store`
- facade 可以保留，但 God file 继续避免回潮。

## 5. 持久化规则

- SQLite 和本地文件仍是 source of truth。
- 浏览器缓存只做体验优化，不做真相存储；RSC 可直接读取的服务端数据不要再单独做客户端 in-memory 数据缓存。
- `src/server/repositories/*` 是持久化入口。
- 旧 `src/server/*-store.ts` 只作为兼容 facade，不再承载新实现。
- `accounts / image files / image conversations / upstream tasks / sync runs / request logs / config` 的新读写逻辑优先落到 repository，而不是写回业务 service 或 route handler。
- 允许保留的客户端 store 类型：
  - 纯 UI 状态，例如图片工作台当前选中会话 / 草稿状态；
  - 浏览器运行中任务状态，例如 active image task；
  - RSC 首屏数据的客户端后续编辑缓存，例如 image conversations。

## 6. Next.js 约定式边界

- 根级必须保留：
  - `src/app/error.tsx`
  - `src/app/loading.tsx`
  - `src/app/not-found.tsx`
- app group 级必须保留：
  - `src/app/(app)/error.tsx`
  - `src/app/(app)/loading.tsx`
  - `src/app/(app)/not-found.tsx`
- 这些文件用于兜住页面级错误、Suspense fallback 和 404，不再依赖默认页或只靠 toast 表达失败。
- 不为了“有 middleware”而增加空 middleware。只有出现鉴权、跨路由 header、rewrite/redirect、locale 等横切需求时才新增 `middleware.ts`。

## 7. 改造优先级

后续默认按下面顺序改：

1. 巨型页面或巨型交互组件
2. server 侧高复杂度编排文件
3. repository / store 边界
4. 共享规则归并
5. 次级页面和通用组件

## 8. 明确不做什么

- 不为了目录好看做大搬家
- 不为了“设计模式完整”硬造抽象层
- 不把所有文件都拆成极细颗粒
- 不把 Next.js 项目硬拆成独立后端服务，除非产品边界真的改变
- 不为了“最佳实践清单完整”添加无业务意义的 noop middleware、空抽象层或重复 facade

## 9. 当前执行标准

以后在这个仓库里，看到以下症状就应该优先重构：

- 单个页面或组件接近或超过 600 到 800 行
- 页面同时管理请求、状态、规则、格式化和复杂 UI
- route handler 里出现明显业务编排
- 同一规则在多个地方重复定义
- server 文件同时混有上游协议、持久化和业务策略

当前推荐的落地模式是：

- 页面 / 组件壳保留在 `app` 或 `components`
- 交互编排进入 `features`
- 服务端能力继续沉到 `server`
- 纯规则集中到 `shared` 或 `features/*-view-model.ts`
